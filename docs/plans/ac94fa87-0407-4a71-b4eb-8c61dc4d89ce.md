## Goal
Make every submission — whether it lands on `POST /api/extract` or `POST /api/extract/async` — return `202 Accepted` + `job_id` in milliseconds and run the pipeline in a background `asyncio` task, so the response can never be killed by a Cloudflare 524 even on a 90-minute source.

## Context

### Bug surface
- The user reports: clicking "Generate" hits `/api/extract` and the proxy returns 524.
- Cloudflare's default `524` is triggered by a request staying open past the edge timeout (~100 s for free tier). The synchronous `/api/extract` blocks for the entire pipeline (ingest + transcribe + extract + narrate), so a 1+ minute source already trips it; a 5-minute source is almost guaranteed to.

### What is still broken / missing
1. **The legacy `POST /api/extract` route is still wired to the synchronous blocking handler.** If the user is on a stale browser tab, a CDN-cached `app.js`, or a deployment that only has the old commit, the click still hits `/api/extract` and still 524s. The 524 will not stop until that URL is also async.
2. **`backend/app/config.py` has duplicate module-level definitions** of `JOB_TIMEOUT_SECONDS`, `JOB_TTL_SECONDS`, `JOB_SWEEP_INTERVAL_SECONDS`, `JOB_WATCHDOG_INTERVAL_SECONDS`. Legal Python but a latent footgun.
3. **No test covers the new async flow end-to-end.** `backend/tests/test_api_jobs.py` still asserts on the legacy SQLite row shape; nothing checks that `POST /api/extract/async` returns 202 in <500 ms, that `GET /api/jobs/{id}` returns the in-memory status, or that `GET /api/jobs/{id}/result` returns 409 until the job is done.
4. **The OpenAPI docstring on `create_app`** still advertises `POST /api/extract` as the primary entry point. Operators reading the auto-generated `/docs` will assume the sync path is the supported one.

### Constraints / runtime assumptions
- Cloudflare edge sits in front of the backend (per the user's 524 report); the longest request the proxy must hold open is now ~5 ms.
- Single-process FastAPI (`uvicorn backend.app.main:app` from `run.sh`); in-memory `JobManager` is fine for v1, no Redis needed.
- The pipeline modules (`pipeline.ingest`, `pipeline.transcribe`, `pipeline.extract`, `pipeline.narrate`, `pipeline.orchestrator`) are unchanged in behavior — they remain CPU-bound and run inside `asyncio.to_thread` from the background task.
- Frontend ships as static files served from `frontend/`; no build step, no service worker, no manifest. Browser cache is the only "deployment" mechanism today.
- The legacy `POST /api/extract` may still be called by external integrations (e.g. the README, curl snippets, third-party scripts); deleting it is a breaking change.

## Assumptions
- The single most likely cause of the user's 524 is that **something is hitting the legacy `POST /api/extract`** — either a stale browser tab, a CDN-cached `app.js`, or an external script. The current working copy of `frontend/app.js` already posts to `/api/extract/async` (line 401), so the frontend itself is fine in the working tree.
- Keeping `/api/extract` as a valid URL is required; making it point to the same async handler (rather than 410-ing or 308-redirecting) is the most forgiving behavior and matches the "still served for backward compat" intent already documented in `app.js:16`. **Open question** (see ambiguities): confirm or pick (b) 308-redirect / (c) 410-Gone / (d) keep sync opt-in.
- The duplicate `config.py` definitions are unintended (a paste-error during the async migration) and can be deduped without behavior change — the second copy has identical defaults to the first.
- The "watchdog / sweeper" background loops in `main.py` start on app startup via the deprecated `on_event("startup")`; they still work, no change needed in this PR.

### Key technical decisions
- **One handler, two URLs.** The cleanest fix is to have `/api/extract` and `/api/extract/async` share the same internal helper. This eliminates the 524 class of bug for any caller, not just the frontend, without breaking anyone who has the old URL bookmarked.
- **No Redis / no DB swap.** The in-memory `JobManager` from `backend/app/jobs.py` is enough for v1. The plan does not add a persistence layer for async jobs; the SQLite `jobs` table continues to be written by the *same* helper inside `_submit_async_job` (so `/api/jobs` and the Recent panel still work).
- **No new frontend code.** The frontend already uses the async flow. The only frontend change is the README-style "What changed?" note — out of scope for this bug fix.

### Risk controls
- **Backward compat risk (medium):** changing the response from 200 → 202 on `POST /api/extract` will break any external script that was parsing the synchronous `LoomaResult` body. The README must call this out in the changelog. If preserving the 200 path is required, alternative is a 308 redirect to `/api/extract/async` — but that breaks non-307-aware clients. The chosen path (alias) is the most forgiving.
- **Sweeper / watchdog race risk (low):** the background loops in `main.py` are scheduled on `app.state` via the deprecated `on_event("startup")`. If a test calls `create_app()` twice in the same process, two sweeper tasks run; not introduced by this PR but easy to trip over. The new test module must use the existing `client` fixture and not call `create_app()` directly.
- **Polling storm risk (low):** if the JS client opens 5 tabs of the same URL, 5 polling loops run for the same job_id; the server is single-process so the cost is trivial, but documented in the Out of Scope section.

## Out of Scope
- Removing the synchronous `/api/extract` route entirely (we keep it as an alias of the async handler; the user's "fix it" was satisfied by neutralizing the 524, not by deleting the URL).
- Adding Redis or any cross-process state for the `JobManager` (the user said "fix it", not "scale it").
- A service-worker or `Cache-Control: no-store` for `app.js` to defeat stale browser caches (worth doing later as a separate ticket; the URL-aliasing in this plan already neutralizes the worst symptom).
- A "What's new in v2" UI banner (no design request, can come in a follow-up).
- Replacing the deprecated `@app.on_event("startup")` with a `lifespan` context manager (separately worth doing, not blocking this bug fix).
- Re-running the full `pytest` suite on the live CI image (out of scope for the fix; the AC-6 module covers the regression surface for the async flow).

## Deployment Suggestions & Platform Detection

**Project type detection:** FastAPI (Python) backend + static HTML/CSS/JS frontend served by the same FastAPI process. Single-process; in-memory job state; no separate frontend build. Local-first tool; today runs on `uvicorn` behind whatever proxy the user puts in front.

**Deployment options** (in increasing effort):
1. **Local / single-host (current state):** `bash run.sh` — works as-is. Suitable for the user's 524 scenario when the proxy timeout is the only concern.
2. **Vercel** — supported for static + serverless Python. Would require splitting the backend into Vercel-style route handlers (one file per endpoint under `api/`) and the frontend into a `public/` directory; the in-memory `JobManager` would not survive across serverless invocations, so the async flow would need a KV/Redis swap (out of scope for this bug fix). The right-hand "Deploy" tab offers a one-click deployment with fully automated CLI-based OAuth authentication.
3. **Cloudflare Pages + Workers** — would mirror option 2 with similar caveats. The 524 the user is seeing comes from this proxy layer; deploying the app *to* that proxy fixes the timeout at the *serverless* tier but reintroduces the multi-process state problem.
4. **Traditional VPS** (DigitalOcean, Fly.io, Hetzner) — recommended path forward: keeps the single-process model intact, allows the in-memory `JobManager` to work as designed, requires only a reverse proxy (nginx/Caddy) with a long backend timeout to bypass Cloudflare.

**Recommendation:** stay on the local/VPS model for v1; revisit serverless deployment in a follow-up PR that adds a KV/Redis `JobManager` implementation.
